# 产品策略：从 ChatBI 工具到本地可信分析工作区

## 定位

### 品类

本地可信分析工作区（Local Trustworthy Analytics Workspace）。

### 定位陈述

面向需要从本地业务数据快速获得可靠结论、又不愿把数据和问题发送到云端的个人经营者、分析师与小型团队，QueryGPT vNext 通过本地 LLM、严格语义编译和一次性分析舱，把自然语言问题转化为可解释、可钻取、可保存和可重放的分析结果。与自然语言转 SQL 聊天框、数据库 IDE 或云端 BI 不同，它不让模型直接写 SQL，不要求账号或服务器，也不把技术执行细节占据主界面。

### 首发楔子

不是“支持最多数据库”，而是把一个本地 SQLite 业务库做到可信、漂亮、低门槛：

- 十分钟内获得第一张可信图表。
- 每个数字都能回到指标定义、时间范围、参数和来源列。
- 用户不懂 SQL 也能完成对比、趋势、拆分、钻取和异常解释。
- 深度预测不离开设备，也不把 Python 放进主应用进程。

## 目标用户与任务

### 主要用户：数据敏感的业务操作者

例如独立开发者、小型电商经营者、财务/运营分析师、咨询顾问。他们通常有 SQLite 导出、桌面业务库或本地数据副本，能描述业务问题，但不愿维护 BI 服务或持续写 SQL。

核心任务：

1. 快速回答“发生了什么”。
2. 理解“为什么发生”。
3. 通过对比、分组、钻取验证结论。
4. 保存一个可复用的结果或看板。
5. 在确有必要时运行趋势、预测或统计分析。
6. 向他人解释数字来自哪里、用了什么定义。

### 次要用户：需要审计能力的分析师

他们会打开 Inspector 检查语义选择、参数化 SQL、血缘、行数、耗时、模型版本和分析代码，但不希望这些细节一直挤占工作区。

### 非目标用户

- 需要多人实时协作、权限中心、云数据仓库或嵌入式 BI 的企业团队。
- 需要写 DDL/DML、管理索引、调试存储过程的数据库管理员。
- 希望任意安装 Python 包、访问网络或把应用当通用 notebook 的数据科学家。
- 需要未经建模就对任意数据库发自然语言 SQL 的用户。

## 竞争格局与取舍

| 类型 | 代表 | 优势 | 结构性缺口 | 我们的回答 |
|---|---|---|---|---|
| 语义层/GenBI | WrenAI、Lightdash、Cube | 强语义模型、指标与关系 | 服务化或工程门槛高；结果工作区不是纯本地桌面核心 | 把严格语义契约、编译证明和本地桌面体验整合为一个安装包 |
| NL-to-SQL Agent | Vanna、DB-GPT | 快速接入、工具丰富 | Python/服务栈重，模型更接近 SQL，UI 往往仍是聊天 | 模型只写 IR；TS 交互，Rust 编译，本地沙箱执行 |
| 数据库 IDE + AI | Chat2DB | 连接器和 SQL 工作流成熟 | 技术细节优先，业务用户学习成本高 | Result Canvas 优先，Inspector 按需，首发不做 IDE |
| 成熟 BI | Metabase、Lightdash | 图表、表格、筛选、看板完整 | 部署和建模较重，AI 与本地隐私不是统一边界 | 本地安装、渐进语义建模、AI 驱动但结果符合 BI 语法 |
| AI Artifact | Claude Artifacts、Vercel Chatbot | 生成体验和结果面板自然 | 不具备 BI 语义、执行证明与数据治理 | 借鉴双栏和 typed artifacts，补齐指标、血缘、钻取、重放 |
| 通用分析沙箱 | E2B/Firecracker 路线 | 隔离强、包生态完整 | 依赖 Linux/KVM 或云基础设施，不符合纯本地 macOS/Windows | 使用有明确边界的一次性 Pyodide 分析舱，不宣称 MicroVM |

## 差异化护城河

### 1. 严格语义编译，而不是更长的 Prompt

LLM 面对的是稳定 ID、闭合集合与版本化 `SemanticQuery`。指标公式、连接、基数、策略和方言由可信模型与 Rust 编译器控制。竞争优势来自协议、诊断、测试语料、数据语义和可重放计划，而不是一个可复制的系统提示词。

### 2. 端到端本地可信链

本地模型、本地数据、本地编译、本地执行、本地持久化、本地字体与本地 Python 包构成同一边界。产品不会把“本地”降格为一个部署选项，也不会偷偷预留云端控制面。

### 3. BI 结果是一级对象

回答不是聊天消息中的图片。结果拥有版本、指标卡、图表、表格、筛选、钻取、血缘、保存状态和重放能力；对话只是构造和修改结果的控制面。

### 4. 深度分析有明确成本与授权

产品不会对每个问题自动跑 Python。只有 SQL 无法完成、输入规模满足预算、代码和资源计划可见、用户明确批准时，才创建一次性分析舱。输出必须回到产品拥有的结构化图表和表格协议。

### 5. 对业务用户简洁，对审计用户透明

同一结果既可以只显示一句结论和一张图，也可以展开完整语义计划、参数化 SQL、血缘、模型版本、耗时与分析代码。复杂度被折叠，而非删除或铺满页面。

## “不做什么”也是策略

- 不追逐连接器数量；SQLite 的完整可信闭环优先于十个半成品连接器。
- 不接受任意 SQL 输入作为核心路径；高级用户也通过 Inspector 查看，而不是绕过编译器。
- 不内置模板市场、云分享或团队权限；它们会改变信任边界。
- 不把 Python notebook 暴露为通用入口；分析舱只执行产品批准的单次任务。
- 不复制完整开源模板；仅使用许可证清晰的 primitives 和独立实现的产品组件。
- 不为旧产品保留兼容层；兼容会把 clean-room 变成双系统维护。

## 产品循环

```text
连接本地数据
  -> 确认业务语义
  -> 提出问题
  -> 获得结果与证据
  -> 钻取/修正
  -> 保存图表或看板
  -> 复用可信上下文提出更深问题
```

每一步都必须提升下一步质量：保存的不是“黄金 SQL”，而是问题、语义 IR、模型修订、计划哈希、结果结构和用户确认的业务含义。

## 关键产品指标

这些指标全部在本地计算并只在本机查看；产品不上传遥测。

### 激活

- `Time to First Trusted Result`：首次启动到第一个通过编译并渲染的结果，中位数目标低于 10 分钟。
- `First-session completion`：选择数据源的用户中完成首次结果的比例，本地诊断目标 ≥ 70%。

### 可信度

- `Compile success after repair`：合法业务问题在最多两次结构化修复内编译成功的比例，基准语料目标 ≥ 90%。
- `Unsafe IR rejection`：恶意/越权/逃生语料被拒绝的比例必须为 100%。
- `Replay identity`：固定模型、语义修订、IR 与策略下，计划哈希和参数化 SQL 一致率必须为 100%。
- `Evidence coverage`：每个成功结果拥有指标定义、时间范围、血缘、计划哈希和执行摘要，目标 100%。

### 使用价值

- 首次结果后的钻取或追问完成率。
- 被保存为图表/看板的成功分析比例。
- Inspector 打开率和复制证据率，用于判断透明度是否可发现但不打扰。
- 分析舱任务中由用户批准、成功完成、被取消和被资源门终止的分布。

### 体验质量

- 从提问到首个结构化进度事件 < 250 ms。
- 取消操作到 producer/DB/分析舱停止的 p95 < 1 s（数据库驱动允许的例外需明确显示）。
- Canvas 首次可用内容不等待完整 Inspector 证据渲染。
- 关键流程键盘完成率 100%，严重可访问性缺陷为 0。

## 发布策略

### 开发顺序

先证明跨平台原生边界，再交付一个 SQLite 的完整纵向切片，然后接本地 LLM，再完善 Workspace，最后加入 Pyodide 分析舱。不能先堆聊天 UI，再在末期补语义和发布。

### 分发

- macOS 提供 arm64 与 x64 安装包；可额外产出 universal 包，但两个原生 ABI 必须各自被加载测试。
- Windows 提供 x64 安装包。
- 正式发布要求签名、公证/信誉检查和干净机器冒烟；开发 CI 产物不等同于发布产物。
- 不提供 Web 版、账号或自动云更新服务。版本更新以用户主动下载、离线可验证安装包为边界。

### 商业边界

纯本地不等于免费或无壁垒。许可可以按设备或版本销售，但激活机制不能要求持续在线、不能成为数据通道，也不能改变产品的离线可用性。具体商业许可不属于首发工程范围。

## 决策门

在扩大范围前，依次回答：

1. 这个能力是否提高可信答案闭环，而不是把产品拉回 IDE？
2. 是否可以在无账号、无云服务、无远程模型下完整工作？
3. 是否能由版本化契约和自动化证据验收？
4. 是否在 macOS arm64/x64 与 Windows x64 都有明确实现？
5. 是否增加新的逃生字符串、第二套 UI/图表/动画体系或宿主执行面？
6. 如果答案需要兼容旧实现，是否应直接拒绝并保持 clean-room？

任何一项不满足，就不进入当前路线图。
